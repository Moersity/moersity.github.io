分布式锁的作用：

​ 在现实世界中，对于某些操作，我们需要保证同时只有一个人在某个地方执行某种操作，举个例子，我们去上厕所🚽，如果有两个人同时进入到了一个坑位，那是会带来不可预料的问题的（emmmm…)

在软件开发过程中也一样，我们想保证程序同时只会有一个进程/线程操作关键数据，在单机的情况下，操作系统系统的锁机制能很好的解决这个问题， 因为很多状态是确定的，单机内部的通信也是很快的，cpu 算错数据出现故障的概率极低，所以单机的锁可以完美解决。

随着数据量的增加，用户量的增加，系统出现了在单机无法满足程序运行的资源（垂直扩张）的时候，我们会想着加机器（水平扩张）来解决，然而由于通信通常是通过网络，而现实互联网是处于一个异步网络的架构的，就向你对这楼上的人喊一声（tmd的大晚上的动静小点), 对方是不一定能及时听到的，你也不能确认对方是否真正听到，即使对方听到了，你也不能保证对方会真正的执行， 所以，锁的实现在复杂网络环境下，变得非常棘手，我们通常会采用第三方软件来保证，常见的有zookeeper, redis, etcd等工具。

​ 扯回来，我们为什么需要分布式的锁，Martin给出了总结：

1. 为了系统的性能，让一个任务不会被多个实例执行，浪费资源。
 2. 为了系统的安全，保证任务按照正确的步骤执行，不会出现文件冲突，内容冲突。

Redis 实现分布式锁

redis 是一个高效的基于内存的数据库，提供了多种数据结构和原子保证，于是很多人就想着用它来实现分布式锁，常见的有以下几种方案：

SETTNX + expire

settnx 是redis 提供的一个语句，意思是如果这个key 不存在，就设置上值， 同时设值的同时对key加一个expire 限制。解锁的时候使用del key 释放。

  1. 安全性：

    同时只会有一个key被设置，保证了安全性

  2. 活性

    有一个超时时间，如果加锁的线程crash，会自动释放，不会存在死等待问题

缺陷及修复：

  1. 使用了settnx + expire 的两条语句，这两条语句的执行并不是原子的，如果只是设置了值而没设置过期时间，会带来死锁问题，解决方案可以把两条语句都放在lua脚本里，脚本的执行是原子的。

  2. 可能出现锁被其他线程释放，如下

    1.1 线程A申请锁成功，执行任务

    1.2 线程A在执行任务时block时间过长，超过了锁的过期时间，导致锁被过期释放

    1.3 线程B发现锁被释放，申请锁成功，开始执行任务

    1.4 线程A恢复，使用del释放了B申请的锁

    ​ 这种情况产生的原因是因为我们释放锁的时候只是简单的删除key，可以采用类似cas的解决方案，加上锁的版本号，因为是在分布式情况下，我们不能简单的产生一个全局自增的id, 所以通常采用uuid的方式（有冲突风险但概率极低），我们将value设置为一个uuid字符串，在释放锁的时候先判断这个锁是不是自己申请的，如果是再删除，不是不做操作，这里注意这两条语句也得放在lua脚本里保证原子。

SET resource_name my_random_value NX PX 30000

通过 set 语句的一些参数，可以实现上面的settnx + expire 的功能， 首先NX的意思是只有key 不存在才设置，PX 就是设置过期时间，bingo, 只用一条语句完成了申请锁，保证了原子性，也不用放在lua脚本里执行了，然后释放锁和之前一样，放在lua脚本里执行判断是否random_value 一样，一样再删除

缺陷：

​ 只适合于redis单机，集群模式无法做到，因为redis机器是异步同步的，如果mater发放了锁但是没有同步到slave, 那么在master宕机的情况下, slave 是不能保证之前发放锁的安全性的

RedLock

​ redlock 是redis 作者设计的用于多master的分布式锁解决方案，具体的步骤如下：

  1. 申请者A首先获取自己的当前时间t1
  2. 申请者A按次序向N个redis master申请锁，申请方式和单机一样，同时使用一样的key和随机值，这里添加了一个timeout时间，这个时间比申请锁自动释放的时间要小，如果申请10s自动释放那么大概timeout时间是5-50ms，一旦超时时间没有得到server 的回复，那么立即向下面的master申请
  3. 申请者A计算当前时间t2, 统计当前回复申请成功的master数量，如果超过了N/2+1的成功，并且t2-t1 大于申请时的锁自动释放时间，那么就当作锁申请成功
  4. 如果申请成功，那么当作当前的锁的有效期为申请的锁自动超时时间-(t2-t1)
  5. 如果申请失败，那么释放之前申请的所以master的锁，释放过程中直接向所有的mater发送删除请求（和单机一样判断随机值是否被更改）
